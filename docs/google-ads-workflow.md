# Google Ads Workflow

End-to-end lifecycle of a Google Ads campaign in this system — from AI-generated draft to live monitoring.

## Overview

```
Agent generates JSON draft
         │
         ▼
clients/[tenant]/ads/google/*.json   (status: "draft")
         │
         ▼ (review in UI or editor)
   status: "approved"
         │
         ▼
scripts/deploy-google-ads.ts
         │
         ▼
  Google Ads API
  (all assets created as PAUSED)
         │
         ▼
   status: "published"  ←  local JSON updated
         │
         ▼
  Enable in Google Ads after review
         │
         ▼
scripts/collect-daily-metrics.ts  (daily)
         │
         ▼
    db/marketing.db  →  UI Alerts Inbox
```

---

## Campaign JSON Format

Campaigns are stored as JSON files in `clients/[tenant]/ads/google/`. Each file is a complete campaign spec including all ad groups, keywords, and ad copy.

```json
{
  "workflow": {},
  "result": {
    "id": "2026-04-16_reformas-alto-padrao-poa",
    "status": "draft",
    "platform": "google_search",
    "objective": "lead_generation",
    "budget_suggestion": "R$50/day",
    "ad_groups": [
      {
        "name": "Reforma Apartamento POA",
        "keywords": [
          "\"reforma apartamento porto alegre\"",
          "[reforma apartamento poa]",
          "empresa de reforma apartamento"
        ],
        "negative_keywords": ["barato", "diy", "curso"],
        "responsive_search_ad": {
          "headlines": [
            "Reforma de Apartamentos POA",
            "Acabamento de Alto Padrão",
            "Orçamento Grátis Hoje"
          ],
          "descriptions": [
            "Especialistas em reformas residenciais de alto padrão em Porto Alegre. Solicite seu orçamento.",
            "Mais de 10 anos reformando apartamentos em POA. Qualidade e pontualidade garantidas."
          ]
        }
      }
    ]
  }
}
```

**Keyword match types** are encoded inline:
- `"palavra-chave"` → Phrase match
- `[palavra-chave]` → Exact match
- `palavra-chave` → Broad match

**`budget_suggestion`** is parsed by the deploy script — any format containing a number works (`R$50/day`, `50`, `R$ 50,00`).

## Status Lifecycle

| Status | Meaning | Where set |
|---|---|---|
| `draft` | Generated by agent, not yet reviewed | Agent |
| `approved` | Reviewed and approved for deployment | UI or manual |
| `published` | Deployed to Google Ads API | Deploy script (auto) |

The deploy script refuses to deploy anything that isn't `approved`.

## Deployment

### Prerequisites

- `brand.json` must have `google_ads_id` (Google Ads customer ID)
- `brand.json` must have `website_url` (used as `final_urls` for all RSAs)
- Campaign status must be `"approved"`
- `.env` credentials must be valid (see `getting-started.md`)

### Running the deploy script

```bash
bun run scripts/deploy-google-ads.ts clients/portico/ads/google/campaign.json
```

Optional: override customer ID if it differs from `brand.json`:

```bash
bun run scripts/deploy-google-ads.ts clients/portico/ads/google/campaign.json 123-456-7890
```

### What the script creates (in order)

1. **Campaign Budget** — one shared budget with the parsed daily amount
2. **Campaign** — Search campaign, created as `PAUSED`
3. **Ad Groups** — one per entry in `ad_groups`, created as `PAUSED`
4. **Keywords** — positive keywords with correct match types
5. **Negative Keywords** — broad match negatives per ad group
6. **RSA (Responsive Search Ad)** — one per ad group, created as `PAUSED`

All assets are created **PAUSED** regardless of the original status. This is intentional — review the campaign in Google Ads before enabling it live. The UI also triggers this script via `POST /api/ads/google/[client_id]/[filename]/deploy`.

### After deployment

- The local JSON `result.status` is updated to `"published"`
- The Google Ads resource names are printed to stdout
- Enable the campaign manually in Google Ads once satisfied with the setup

## Live Metrics

The UI fetches live data directly from the Google Ads API — no caching, always current.

**Route:** `/[tenant]/ads/google/live/[campaign_id]`

Data available:
- Impressions, clicks, cost, conversions
- CTR, CPA, interaction rate
- Breakdown by ad group with individual metrics
- 30-day history chart

**API library:** `google-ads-api` v23 via `scripts/lib/ads.ts`. Use `getCustomer(customerId)` to get a client — never instantiate `GoogleAdsApi` directly.

## Daily Monitoring

Once a campaign is live, `scripts/collect-daily-metrics.ts` runs daily and:

1. Queries all non-removed campaigns for the tenant
2. Fetches the previous day's metrics
3. Evaluates thresholds from `brand.json → ads_monitoring`
4. Writes to `daily_metrics` table (all campaigns, all alert levels)
5. Writes WARN and CRITICAL events to `alert_events` (surfaced in UI inbox)

See `docs/monitoring-system.md` for full details on alert logic and configuration.

## Reporting

After collecting daily data, you can generate markdown reports for deeper analysis:

**Manual report via agent:**

Open Claude Code and invoke the monitoring or weekly-report agent. The agent reads from SQLite and writes to `clients/[tenant]/reports/`.

**Monthly consolidation:**

```bash
bun run scripts/consolidate-monthly.ts portico
```

Aggregates `daily_metrics` into `monthly_summary` for the current month. Run at end of month or before generating a monthly report.

**Report naming conventions:**

| Type | Filename pattern | UI color |
|---|---|---|
| Audit | `google-ads-audit-YYYY-MM-DD.md` | Amber |
| Search campaign | `google-ads-search-YYYY-MM-DD.md` | Blue |
| Weekly | `weekly-YYYY-MM-DD.md` | Emerald |
| Monthly | `google-ads-YYYY-MM.md` | Violet |
| Alert | `*alert*.md` | Red |

## Testing Without Deploying

Use the test scripts to inspect account data without making any changes:

```bash
# Verify connection
bun run scripts/test-ads-connection.ts <customer-id>

# List campaigns with metrics
bun run scripts/test-query.ts <customer-id> <campaign-id>

# List ad groups for a campaign
bun run scripts/test-query-ag.ts <customer-id> <campaign-id>

# 30-day history for a campaign
bun run scripts/test-query-history.ts <customer-id> <campaign-id>
```

These are read-only and safe to run at any time.

## Key Rules

- **Never modify live campaigns autonomously.** Always show the proposed change, wait for explicit confirmation, then execute.
- **Customer IDs and campaign IDs are never committed.** They live in `brand.json` (gitignored) and `.env`.
- **Deploy always creates PAUSED assets.** Enable manually after review in Google Ads.
- **Re-running the deploy script on the same JSON is safe** only if you delete the previously created assets first — the script always creates new resources.
